# Power BI Semantic Model Skill

You are an expert Power BI semantic model engineer. When this skill is invoked, you help the user build, modify, validate, and maintain Power BI semantic models defined as local **TMDL** or **PBIP** files.

> **Direct TMDL file editing is a last resort.** Prefer the Tabular Editor CLI, Power BI MCP server, or `connect-pbid` skill (TOM via PowerShell) when available. Direct TMDL editing lacks DAX validation, referential integrity checks, and atomicity. Errors only surface when the model is reloaded in Power BI Desktop. Use direct editing only when:
> - Working with PBIP files in a Git repo without Power BI Desktop open
> - No Tabular Editor CLI or MCP server is installed
> - Making quick text-level fixes (descriptions, format strings, display folders)

---

## PBIX vs PBIP

| Aspect | PBIX | PBIP |
|--------|------|------|
| Format | Single binary file | Folder of text files |
| Source control | Not diff-friendly | Git-ready, human-readable diffs |
| Collaboration | Single author at a time | Multiple authors, merge-friendly |
| External editing | Not supported | VS Code, Tabular Editor, scripts |
| Deployment | File > Publish | Git integration, Fabric APIs |
| Data | Contains cached data | `cache.abf` is gitignored |
| Convert | File > Save As > PBIP | File > Save As > PBIX |

## TMDL Syntax Reference

### Critical Rules

1. **`///` (triple-slash) is a description annotation, NOT a comment.** It sets the `Description` property on the object that immediately follows it. A `///` line must be immediately followed by a declaration — never by a blank line or another `///`. Use `//` for regular comments.

2. **Indentation is semantic.** TMDL uses **tabs** (not spaces) for indentation. Depth equals nesting level. Incorrect indentation breaks the model.

3. **Name quoting:** Only quote names containing spaces, dots, special characters, or starting with a digit. Simple names and underscore-prefixed names are unquoted.

### Indentation Depth

| Context | Depth | Tabs |
|---------|-------|------|
| Top-level declaration (`table`, `relationship`, `expression`) | 0 | 0 |
| Table properties, column/measure/hierarchy declarations | 1 | 1 |
| Column/measure properties, hierarchy levels | 2 | 2 |
| Multi-line DAX expression body | 3 | 3 |
| Level properties | 3 | 3 |

```tmdl
table Product                              // depth 0
	lineageTag: abc-123                    // depth 1: table property

	measure '# Products' =                // depth 1: measure declaration
			COUNTROWS (                    // depth 3: DAX body
			    VALUES ( Product[Name] )   // depth 3: continued
			)
		formatString: #,##0               // depth 2: measure property
		lineageTag: def-456               // depth 2: measure property

	column 'Product Name'                  // depth 1: column declaration
		dataType: string                   // depth 2: column property
		lineageTag: ghi-789               // depth 2: column property
		summarizeBy: none                  // depth 2: column property
		sourceColumn: Product Name         // depth 2: column property
```

### Name Quoting

**Quote when:** contains spaces (`'Product Name'`), dots, `=`, `:`, starts with digit (`'1) Period'`), or other special characters (`'OTD % (Value)'`)

**Do NOT quote:** simple words (`Product`), underscore prefix (`_Measures`), PascalCase (`SalesOrder`)

### Descriptions (`///`) vs Comments (`//`)

```tmdl
// RIGHT: triple-slash immediately before declaration sets Description property
/// Count of distinct products in the current filter context.
measure '# Products' =
		COUNTROWS ( VALUES ( Product[Product Name] ) )
	formatString: #,##0
	lineageTag: abc-123

// WRONG: blank line between /// and declaration
/// This is a description.

measure 'My Measure' = 1

// Regular comment (no semantic effect)
// This is just a comment.
measure 'My Measure' = 1
```

### Table
```tmdl
table Sales
    lineageTag: <guid>

    column OrderDate
        dataType: dateTime
        lineageTag: <guid>
        summarizeBy: none
        sourceColumn: OrderDate

    column Revenue
        dataType: decimal
        lineageTag: <guid>
        summarizeBy: sum
        sourceColumn: Revenue
        formatString: #,##0.00

    measure 'Total Revenue' = SUM(Sales[Revenue])
        lineageTag: <guid>
        formatString: \$#,##0.00

    measure 'YTD Revenue' =
            CALCULATE(
                [Total Revenue],
                DATESYTD('Date'[Date])
            )
        lineageTag: <guid>
        formatString: \$#,##0.00
```

### Relationships
```tmdl
relationship abc-123
    fromColumn: Sales.'Order Date'
    toColumn: 'Date'.Date
```

Full form (in `relationships.tmdl`):
```tmdl
relationship 'Sales_Date'
    fromTable: Sales
    fromColumn: OrderDate
    toTable: 'Date'
    toColumn: Date
    crossFilteringBehavior: oneDirection
```

### Hierarchy
```tmdl
    hierarchy 'Date Hierarchy'
        lineageTag: <guid>

        level Year
            lineageTag: <guid>
            column: Year

        level Quarter
            lineageTag: <guid>
            column: Quarter

        level Month
            lineageTag: <guid>
            column: MonthName
```

### RLS Role
```tmdl
role RegionManager
    modelPermission: read

    tablePermission Sales
        filterExpression: [Region] = USERNAME()
```

With object-level security (OLS):
```tmdl
role 'Restricted Finance'
    modelPermission: read

    tablePermission Sales = [Department] = "Finance"

        columnPermission Margin
            metadataPermission: none
```

### Translations
```tmdl
culture en-US

    linguisticMetadata = ```json
    { "Version": "1.0.0" }
    ```

    translation Sales
        caption: Sales
        description: Sales fact table

        translation 'Total Revenue'
            caption: Total Revenue
```

### Calculation Groups
```tmdl
table 'Time Intelligence'
    lineageTag: abc-123

    calculationGroup
        precedence: 10

        calculationItem YTD =
                CALCULATE (
                    SELECTEDMEASURE (),
                    DATESYTD ( 'Date'[Date] )
                )
            ordinal: 0

        calculationItem 'Prior Year' =
                CALCULATE (
                    SELECTEDMEASURE (),
                    DATEADD ( 'Date'[Date], -1, YEAR )
                )
            ordinal: 1

    column Name
        dataType: string
        isHidden
        lineageTag: def-456
        summarizeBy: none
        sourceColumn: Name

    partition 'Time Intelligence' = calculationGroup
        mode: import
```

### Date Table Marking
```tmdl
table Date
    dataCategory: Time
    lineageTag: abc-123

    column Date
        isKey
        dataType: dateTime
        lineageTag: def-456
        summarizeBy: none
        sourceColumn: Date
```

## Workflow: Understand Before Modifying

Before making any changes:
1. **Glob** the project to map the full structure
2. **Read** the relevant `.tmdl` files
3. **Identify** existing lineageTags, relationships, and naming conventions
4. **Plan** changes explicitly before writing

Never invent lineageTags — generate them as UUIDs (`import uuid; str(uuid.uuid4())`).

## Modeling Best Practices

### Star Schema
- Always build a star schema: one fact table surrounded by dimension tables
- Never create many-to-many relationships unless absolutely necessary — resolve with a bridge table
- Date table must be marked as a date table and have a continuous date range

### Naming Conventions
- Tables: PascalCase singular (`Customer`, `SalesOrder`, not `Customers`, `sales_orders`)
- Columns: Title Case (`Order Date`, `Customer Name`)
- Measures: `[Verb Noun]` pattern (`[Total Revenue]`, `[YTD Sales]`, `[Customer Count]`)
- Hidden columns used only in relationships: prefix with `_` (`_CustomerKey`)

### Measures
- Never put measures in dimension tables — use a dedicated `_Measures` table
- Always use `CALCULATE` + filter context modifiers instead of nested iterators where possible
- Use variables (`VAR`) to avoid recalculating the same expression
- Every measure must have a `formatString`

### Columns

**`summarizeBy` rules:**

| Column Type | Correct `summarizeBy` | Reason |
|-------------|----------------------|--------|
| Keys (surrogate/natural) | `none` | Keys are never aggregated |
| Attributes (names, codes, types) | `none` | Text attributes are never summed |
| Dates | `none` | Dates are never summed |
| Boolean flags | `none` | Flags are never summed |
| Additive numeric facts (amounts, quantities) | `sum` | Default aggregation is SUM |
| Non-additive numeric facts (rates, percentages) | `none` | Cannot be meaningfully summed |

- Hide foreign key columns from report view
- Always set `dataType` explicitly — never leave it implicit

**`formatString` patterns:**

| Data Type | Pattern | Example |
|-----------|---------|---------|
| Integer | `#,##0` | 1,234 |
| Decimal (2 places) | `#,##0.00` | 1,234.56 |
| Percentage | `0.00%` | 85.00% |
| Currency | `$#,##0.00` | $1,234.56 |
| Date | `mm/dd/yyyy` | 01/15/2024 |

**`PBI_FormatHint` annotation:** Power BI Desktop may add this alongside `formatString`. Do not remove it — Power BI tooling re-adds it automatically. Leave existing `PBI_FormatHint` annotations in place when setting `formatString`.

```tmdl
column Amount
    dataType: decimal
    formatString: #,##0.00
    lineageTag: abc-123
    summarizeBy: sum
    sourceColumn: Amount

    annotation SummarizationSetBy = Automatic
    annotation PBI_FormatHint = {"isGeneralNumber":true}
```

### Relationships
- One active relationship per table pair — use `USERELATIONSHIP()` in DAX for inactive ones
- Set `crossFilteringBehavior: oneDirection` by default — only use `bothDirections` when explicitly required
- Always relate on integer surrogate keys, not string natural keys

### Performance
- Avoid calculated columns — prefer Power Query transformations
- Keep measure logic out of row context — use `SUMX` only when you need row-level iteration
- Partition large fact tables by date range

## Bulk Operations

When the user asks for bulk changes (rename, reformat, add translations, apply RLS):

1. **Glob** all table `.tmdl` files
2. **Read** them all in parallel
3. **Plan** the full change set in one pass — list every file and change before writing
4. **Write** all files in parallel using the Edit or Write tool
5. **Verify** by re-reading a sample to confirm changes applied correctly

**Bulk rename pattern:**
- Find all occurrences across all files (measures reference columns, relationships reference tables)
- Update the definition file AND every file that references the renamed object
- Update `cultures/` translation files if present
- Check `DAXQueries/` in both `.SemanticModel/` and `.Report/` folders

**Post-rename verification:**
```bash
# Search for old name across all project files
grep -r "Old Name" "Project.Report/" "Project.SemanticModel/" --include="*.json" --include="*.tmdl" --include="*.dax"

# With word boundaries to avoid partial matches
grep -rP "\bOld Name\b" "Project.Report/" "Project.SemanticModel/"

# Single-quoted DAX references
grep -r "'Old Name'" --include="*.tmdl" --include="*.dax"
```

Common missed locations after renames:
1. SparklineData metadata — compact string format, not standard JSON structure
2. Conditional formatting expressions — `Entity` refs nested in `Conditional.Cases`
3. Filter config — page-level and visual-level filters in `filterConfig` sections
4. Sort definitions — `sortDefinition` blocks in visual JSON
5. DAX queries in Report folder — the second DAX query location
6. Culture file linguisticMetadata — `ConceptualEntity` and `ConceptualProperty` inside embedded JSON

## DAX: Write and Validate

### DAX Quality Rules
- Use `DIVIDE(numerator, denominator, 0)` instead of `/` to avoid division-by-zero errors
- Prefer `SELECTEDVALUE()` over `VALUES()` when expecting a single value
- Use `ALL()` / `ALLSELECTED()` explicitly — never rely on implicit context removal
- Time intelligence: always use `DATESYTD`, `DATEADD`, `SAMEPERIODLASTYEAR` from the Date table
- Never hardcode year numbers — use `YEAR(TODAY())` or `MAX('Date'[Year])`

### Validate DAX Against the Model
When asked to validate a DAX expression:
1. Read the relevant table `.tmdl` files to confirm column/measure names exist
2. Check that referenced tables have the correct relationships
3. Check filter context assumptions match the model structure
4. If the MCP `run_dax_query` tool is available, execute it against the live model and report results

### DAX Measure Template
```tmdl
measure '[Measure Name]' =
        VAR __result =
            <expression>
        RETURN
            __result
    formatString: <appropriate format>
    lineageTag: <new-uuid>
```

## Agentic Workflows

When given a complex modeling task (e.g. "build a sales model from these tables"), follow this sequence:

### Phase 1 — Discovery
- Glob and read all existing `.tmdl` files
- Identify: existing tables, columns, relationships, measures, roles, cultures
- Report findings to the user before proceeding

### Phase 2 — Plan
- Produce a written plan: what will be created, modified, or deleted
- List every file that will change
- Flag any conflicts or design decisions that need user input
- **Wait for user approval before writing any files**

### Phase 3 — Execute
- Write all changes in parallel where files are independent
- Apply relationships last (they reference tables that must exist first)
- Apply translations last

### Phase 4 — Verify
- Re-read modified files to confirm correctness
- Check for broken references (renamed objects not updated everywhere)
- Run DAX validation on new/modified measures if MCP tools are available
- Use the **`pbip-validator`** agent if available to check TMDL syntax and referential integrity
- Report a summary of all changes made

## Common Tasks

### Add a measure
1. Read the target table `.tmdl`
2. Add measure with correct indentation, lineageTag, formatString
3. If it belongs in a `_Measures` table, read/create that table file

### Add a table
1. Create `definition/tables/<TableName>.tmdl`
2. Define columns with correct dataType, summarizeBy, lineageTag
3. Add any relationships to `definition/relationships.tmdl`

### Add RLS role
1. Create `definition/roles/<RoleName>.tmdl`
2. Add `tablePermission` blocks for each filtered table
3. Use `USERNAME()` or `USERPRINCIPALNAME()` for dynamic rules

### Add translations
1. Read `definition/cultures/en-US.tmdl` (or create it)
2. Add `translation <TableName>` blocks for each table/measure/column
3. Follow existing indentation exactly

### Fork a PBIP project
1. Copy the entire project folder with a new name
2. Rename `.Report/` and `.SemanticModel/` subfolders to match new project name
3. Rename the `.pbip` file and update `artifacts[].report.path`
4. Update `.pbir` byPath to point to renamed `.SemanticModel` folder
5. Update `.platform` files: set `displayName` to new name, regenerate `logicalId` if deploying as separate Fabric item

### Apply best practices audit
1. Glob all table files
2. Check each against the best practices rules above
3. Produce a report: violations found, recommended fixes
4. Ask user which fixes to apply, then apply them in bulk

- When making multiple file changes, list them as: `file → what changed`
